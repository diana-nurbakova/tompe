# Explanation Generation

How ToM-informed feedback explanations are produced, sourced, cached, and
surfaced to students. Covers all three explanation layers, the cache-first
lookup for the per-type layers, how `mt_system` drives that lookup, the
end-to-end data flow into the student UI, and a worked LLM-call example.

Spec reference: system spec v1.1 §5.4.

---

## 1. The three layers

Explanations are generated as a pipeline stage
(`segment → inject / detect → explanations`) and come in three layers with
different scopes:

| Layer | Scope | Purpose | Source |
|-------|-------|---------|--------|
| **Layer 1 — contrastive** | per error *instance* | Understand *this* error from MT / author / reader perspectives | LLM, per instance (no cache) |
| **Layer 2a — popular science** | per error *type* | How MT systems make this *class* of error, no jargon | JSON cache → LLM fallback |
| **Layer 2b — technical** | per error *type* | Technical NLP depth (progressive disclosure) | JSON cache → LLM fallback |

All three are defined in
[`src/tompe/pipeline/explanation_generator.py`](../src/tompe/pipeline/explanation_generator.py);
their prompts and JSON schemas live in
[`src/tompe/pipeline/_injection_prompts.py`](../src/tompe/pipeline/_injection_prompts.py).

Each layer maps to a Pydantic model on the error object
([`src/tompe/schemas/error.py`](../src/tompe/schemas/error.py)):

| Layer | Error attribute | Model | Fields |
|-------|-----------------|-------|--------|
| 1 | `explanation` (required) | `ContrastiveExplanation` | `mt_interpretation`, `actual_meaning`, `reader_impact`, `correction_rationale` |
| 2a | `system_behavior` (optional) | `SystemBehaviorExplanation` | `error_mechanism`, `architectural_cause`, `pattern_generalization`, `mt_system_specific` |
| 2b | `technical_explanation` (optional) | `TechnicalExplanation` | `technical_description`, `key_concepts[]`, `references[]` |

Both `InjectedError` (controlled pathway) and `DetectedError` (authentic
pathway) carry all three attributes.

## 2. Sources — where each layer's content comes from

### Layer 1 (contrastive) — LLM, instance-specific

`generate_contrastive_explanation()` always calls the LLM (no cache). The
prompt is fed the **full instance context**:

- source text and reference translation,
- the original (correct) span vs. the injected/erroneous span,
- taxonomy label (`primary_tag > error_type`), severity, and `brief_explanation`.

In practice Layer 1 is usually **not** freshly generated by this function: the
**error-injection step** already emits a contrastive `explanation` object (via
`STEP2_RESPONSE_SCHEMA` in the injector). `build_item` only calls the LLM here
when injection's explanation is missing or a placeholder (under 4 words —
`_is_placeholder_explanation`). See
[`item_builder.py`](../src/tompe/pipeline/item_builder.py) lines ~238–255.

### Layer 2a / 2b — cache-first, then LLM

Because 2a and 2b are per-error-*type* (not per-instance), a committed JSON
cache is consulted **before** any LLM call:

- `data/codebook/layer2a_explanations.json`
- `data/codebook/layer2b_explanations.json`

Lookup key is `(primary_tag, error_type, mt_system)` via `_lookup_explanation()`.
A **system-specific entry wins over a wildcard** entry (`mt_system: null`). On a
cache miss — or a malformed entry — generation falls back to the LLM. The LLM
prompt for 2a/2b receives only `primary_tag`, `error_type`, `mt_system`, and
`brief_explanation` — **not** the source/reference text, since these
explanations describe the error *class*, not the instance.

> **Note:** the explanation generators do **not** read the error codebook
> (`error_codebook_fr_en.json` and its drafts/stubs). The codebook supplies the
> taxonomy used upstream during injection/detection; only the resulting
> `primary_tag` / `error_type` / `brief_explanation` strings are threaded into
> the explanation prompts. The Layer-2 caches are a separate store that merely
> happens to live under `data/codebook/`.

## 3. `mt_system` — how the lookup key is chosen

`mt_system` is a free-form label on the item and the third component of the
Layer-2 cache key. It flows as follows:

- `build_item(mt_system="reference_perturbed")` is the default.
- **CONTROLLED pathway** (error injection): stays `"reference_perturbed"`
  unless the caller passes a real label (e.g. `"google_translate"`,
  `"deepseek_v3"`). The teacher preview passes its chosen `system_name` straight
  through.
- **AUTHENTIC pathway**: if still the default, it is overwritten from the real
  `mt_output.mt_system` (`item_builder.py` ~lines 230–231).
- The value reaches `generate_layer2a_explanation(mt_system=...)` /
  `generate_layer2b_explanation(mt_system=...)` and into the cache lookup.

The LLM-fallback prompt also branches on it:
`build_layer2a_user_prompt` maps `gpt4 / claude / ollama / deepseek / together`
→ *"a general-purpose LLM"*, everything else → *"a dedicated neural MT system"*.

> Because every committed cache entry currently uses `mt_system: null`
> (wildcard), the system label does **not** yet change which cache entry is
> picked — it only affects the LLM-fallback wording. System-specific cache
> entries can be added later and will take precedence automatically.

## 4. End-to-end data flow

```
                build_item / build_batch            (pipeline)
                generate_all_explanations            (teacher preview)
                        │
                        ▼
   error object:  .explanation            (Layer 1)
                  .system_behavior         (Layer 2a)
                  .technical_explanation   (Layer 2b)
                        │
                        ▼
   feedback service  (services/feedback.py)   ← reads attributes only,
                        │                        does NOT generate
                        ▼
   card["layer1"] / card["layer2a"] / card["layer2b"]
                        │
                        ▼
   student UI  (interfaces/student_app.py)  — progressive disclosure
```

- **Card assembly:** `feedback.py` (~lines 137–164) reads
  `gt_err.explanation` / `.system_behavior` / `.technical_explanation` and
  flattens them into `layer1` / `layer2a` / `layer2b` dicts on each error card.
  The feedback service does **not** import `explanation_generator`; it only
  reads what was attached upstream.
- **UI rendering (progressive disclosure)** — `student_app.py` (~lines 312–349):
  - **Layer 1** → always-visible blue *"Explanation:"* box, all four fields.
  - **Layer 2a** → collapsed `<details>` *"How It Works"*; shows
    `error_mechanism` + italicized `pattern_generalization`.
  - **Layer 2b** → collapsed `<details>` *"Under the Hood"*; shows
    `technical_description` + `key_concepts`.
  - Fields the UI currently does **not** display: 2a's `architectural_cause`
    and `mt_system_specific`; 2b's `references`. (Stored, available, not shown.)

### Who attaches each layer in the live paths

| Layer | Attached by |
|-------|-------------|
| 1 | Error-injection step (`inject_errors_reference_based`); fallback `generate_contrastive_explanation` in `build_item`; `generate_all_explanations` in teacher preview |
| 2a | `generate_layer2a_explanation` via `build_item` (`generate_layer2a_explanations=True`) or teacher preview |
| 2b | `generate_layer2b_explanation` via `build_item` (`generate_layer2b_explanations=True`) or teacher preview (`include_layer2b=True`) |

## 5. Layer 2b wiring (now active)

Layer 2b was previously fully implemented (generator, schema, model, feedback
card, UI panel) but **never invoked** — `build_item` handled only layers 1 + 2a,
and the teacher preview called `generate_all_explanations(include_layer2b=False)`.
It is now wired into both live paths:

- `build_item` / `build_batch` accept a `generate_layer2b_explanations: bool`
  flag (default `False`), mirroring the Layer 2a handling: cache-first → LLM,
  result attached to `err.technical_explanation` and aggregated into the new
  `AssessmentItem.explanations_layer2b` list.
- The teacher preview now calls
  `generate_all_explanations(..., include_layer2b=True)`, attaches
  `err.technical_explanation`, and populates `explanations_layer2b`.
- `AssessmentItem` gained an `explanations_layer2b: list[TechnicalExplanation]`
  field for symmetry with layers 1 and 2a.

`feedback.py` and `student_app.py` already consumed `technical_explanation` /
`card["layer2b"]`, so no changes were needed there — the panel now renders
whenever an item is built with Layer 2b enabled.

Like Layer 2a, Layer 2b is **opt-in** (default `False`) to keep `build_item`
cheap; enable it for advanced or progressive-disclosure items.

## 6. Worked example — a Layer 1 LLM call

Injected `MISTRANSLATION/false_cognate`: French *"Il a éventuellement accepté"*
rendered *"He eventually accepted"* (correct sense: "possibly/maybe").
`generate_contrastive_explanation` issues `llm_client.complete_json(...)` at
`temperature=0.5`.

**System prompt** (`SYSTEM_PROMPT_CONTRASTIVE`):

```
You are an expert in translation pedagogy and Theory of Mind applied to machine
translation post-editing training. Your task is to generate a contrastive
explanation for a specific translation error, helping students understand the
error from multiple perspectives (MT system, author, reader).
Respond with valid JSON only.
```

**User prompt** (`build_contrastive_user_prompt`):

```
Source text: Il a éventuellement accepté la proposition.
Reference translation: He possibly accepted the proposal.

Error details:
- Original (correct): "possibly"
- Error (injected): "eventually"
- Type: MISTRANSLATION > false_cognate
- Severity: major
- Context: 'éventuellement' is a faux ami — it means "possibly", not "eventually"

Generate a contrastive explanation with these fields:
- mt_interpretation: What the MT system likely "thought" (1-2 sentences)
- actual_meaning: What the source actually means (1-2 sentences)
- reader_impact: How a reader would misinterpret this (1-2 sentences)
- correction_rationale: Why the correct translation is correct (1-2 sentences)
```

**Schema enforced** (`CONTRASTIVE_EXPLANATION_SCHEMA`): object requiring all
four string fields, `additionalProperties: false`. The validated JSON becomes a
`ContrastiveExplanation`, stored as `err.explanation`, rendered in the blue box.

For **Layer 2a/2b** the shape is the same but the cache is consulted first: a
`false_cognate` lookup hits the committed wildcard entry and returns it **with
no LLM call**; only an uncached type fires the LLM with the corresponding
`SYSTEM_PROMPT_LAYER2A` / `SYSTEM_PROMPT_LAYER2B` and
`build_layer2a_user_prompt` / `build_layer2b_user_prompt`.

## 7. Committed cache contents (current)

Each cache currently holds **two wildcard entries** (`mt_system: null`):
`MISTRANSLATION/false_cognate` and `GRAMMAR/agreement_gender`. Every other
taxonomy entry is a cache miss → LLM fallback.

### Layer 2a — `data/codebook/layer2a_explanations.json`

**`MISTRANSLATION/false_cognate`**
- *error_mechanism:* "False cognates (faux amis) are word pairs that share
  orthographic form across languages but diverge in meaning. MT systems often
  pick the surface-similar word because it has a high translation probability in
  the bilingual lexicon, even when the contextual sense is wrong."
- *architectural_cause:* "Subword vocabularies and word-alignment statistics in
  NMT bias the model toward orthographically similar substitutions. The decoder
  weighs token-level cross-entropy more heavily than semantic faithfulness when
  the surface forms align."
- *pattern_generalization:* "Expect this error in any FR↔EN pair where Latinate
  roots have drifted (e.g. 'actually'/'actuellement', 'sensible'/'sensible',
  'eventually'/'éventuellement')."
- *mt_system_specific:* "Both Google Translate and DeepL exhibit this; large-LLM
  translators (DeepSeek-V3, GPT-4) reduce but do not eliminate it because the
  same statistical regularities exist in their pre-training corpora."

**`GRAMMAR/agreement_gender`**
- *error_mechanism:* "French gender agreement requires the article, adjective,
  and past participle to match the noun's grammatical gender. MT systems
  sometimes pick the wrong gender form when the noun is rare or when the
  antecedent is ambiguous in English (which has no gender)."
- *architectural_cause:* "Cross-lingually, English provides no gender signal for
  most nouns, so the model must rely on French monolingual priors. When the
  target noun has low frequency or competing forms, the priors are weak and
  gender errors leak through."
- *pattern_generalization:* "Most common when (a) the French noun is feminine
  but the English source is gender-neutral, (b) the noun is rare, or (c) gender
  disagreement spans several tokens."
- *mt_system_specific:* "Statistical / NMT systems show this more than large
  LLMs. LLM translators still err on rare feminine nouns and on long-distance
  agreement."

### Layer 2b — `data/codebook/layer2b_explanations.json`

**`MISTRANSLATION/false_cognate`**
- *technical_description:* "Cross-lingual lexical interference between
  orthographically aligned but semantically divergent word pairs. Subword
  tokenisation (BPE / SentencePiece) frequently yields shared subword units
  across the language pair, biasing the encoder–decoder attention toward the
  high-probability surface-form mapping. The phenomenon is amplified when
  training data contains parallel sentences with literal renderings."
- *key_concepts:* false cognate · lexical transfer · BPE shared subwords ·
  translation memorisation · cross-lingual word embedding bias
- *references:* Frunza & Inkpen (2009); Carpuat & Wu (2007)

**`GRAMMAR/agreement_gender`**
- *technical_description:* "Cross-lingual divergence in grammatical gender
  features. Where the source language under-specifies gender (English), the
  decoder must infer it from target-side monolingual priors and from the
  alignment of the target noun. Errors arise when (a) the noun is low-frequency
  in training data, (b) gender agreement crosses a long dependency, or (c) the
  antecedent is structurally distant."
- *key_concepts:* morphological agreement · grammatical gender · long-distance
  dependency · decoder LM prior · feature underspecification
- *references:* Vanmassenhove et al. (2018); Stanovsky et al. (2019)

### Extending the cache

Add objects to the `entries[]` array with the keying fields
(`primary_tag`, `error_type`, optional `mt_system`) and all content fields
non-empty. Omit `mt_system` (or set it `null`) to match any system; set it to a
specific label to override the wildcard for that system.
